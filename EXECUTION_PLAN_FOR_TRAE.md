# AetherCrawler 执行方案（供 Trae 执行）

> 基于 OPTIMIZATION_ROADMAP.md 整理。原文档只列了"做什么"，这份补充"怎么做、会踩什么坑、坑了怎么办"。按顺序执行，不要跳步。

---

## 执行前必读：4 条铁律

1. **每完成一个任务，先在本地跑通再 commit。** 不要让 Trae 连续完成多个任务后一次性提交——出问题难定位是哪一步炸的。
2. **每个任务开始前，先 `git checkout -b` 开新分支。** 哪怕你是个人项目，分支隔离能让你随时回滚，不会把 main 改坏。
3. **所有 API Key、Webhook URL 必须进 `.env`，不能硬编码进代码，不能 commit 进 git。** 飞书 Webhook URL 本质上是一个**不需要认证就能调用的接口**——一旦这串 URL 出现在 git 历史里（哪怕后来删除了，历史记录还在），任何拿到它的人都能往你的飞书群发任意消息。这是开源项目，密钥泄露是真实风险，不是危言耸听。
4. **每个自动化功能上线前，先想清楚"怎么一键关掉它"。** 自动化的反面是：出问题时扩散速度也是自动的。定时任务、Webhook推送上线前，确保 `.env` 里有一个 `ENABLE_AUTO_NOTIFY=true/false` 这样的开关，半夜炸消息时改一个值重启服务就能止血，不需要紧急改代码。

## 阶段零：开工前的环境验证（15分钟，但能省后面几小时排查成本）

**这一步经常被跳过，但跳过的代价很高。**

```
给 Trae 的指令：

1. 在改任何代码之前，先确认现有项目能在本机干净环境跑起来：
   docker compose down -v && docker compose up --build
2. 跑通后，手动触发一次现有的分析功能（BTC情绪分析），
   确认能拿到正常输出，记录这次输出结果（截图或保存日志）
3. 记录当前 requirements.txt / package.json 里所有依赖的版本号
   （后续如果新功能引入依赖冲突，能知道改动前的基准状态是什么）
```

**为什么这步不能省**：如果后续任务1.1做完之后发现"AI分析结果变奇怪了"，你需要知道这是不是NewsAPI改动引入的问题，还是项目本来就有这个问题、只是你之前没注意到。没有"改动前的基准"，排查时会怀疑一切，浪费大量时间。

## 阶段一：P0 任务（今天就能做完）

### 任务 1.0（新增，建议放在最前面做）：给 AI 推理引擎加一层"安全带"

**为什么要把这个排在最前面**：原方案里 NewsAPI 和定时任务都是在"用"AI推理引擎，但都没管它本身够不够稳。一旦上了定时任务，调用次数会立刻上升，这时候才发现千问调用没有重试、没有格式校验，代价更大。**先把这层基础打好，后面的任务才不会建立在沙子上**。

```
给 Trae 的指令：

1. 找到项目里调用通义千问(qwen-plus)的核心函数，加上：
   - 超时设置（建议15-20秒，避免一次调用卡死整个任务队列）
   - 失败重试（最多重试2次，间隔3秒，仍失败则记录日志并返回明确的
     "本次分析失败"状态，而不是让程序崩溃或返回空值）

2. 加一层输出格式校验：
   - 千问偶尔会不严格按照prompt要求的JSON格式输出
     （比如多加了markdown代码块标记，或者漏了某个字段）
   - 用 try/except 包裹JSON解析，解析失败时记录原始返回内容到日志，
     方便后续排查是prompt的问题还是模型抽风

3. 加一个简单的成本监控：
   - 每次调用后记录本次消耗的 token 数（通义千问API返回里通常带这个字段）
   - 写入日志或简单存到 Redis，方便你后续观察"加了定时任务之后
     一天大概花多少token/多少钱"，避免月底账单吓一跳
```

**可能踩的坑 + 最优解**：

| 风险点 | 具体表现 | 解决方案 |
|---|---|---|
| 一次调用卡死，定时任务整体延迟 | 没设超时，网络抖动时一次调用可能挂起几分钟，连带后面的任务排队 | 必须设置显式超时，超时后直接判定为失败，走重试逻辑 |
| 输出格式不稳定导致前端崩溃 | 千问返回的JSON多了个逗号或者用markdown包了一层，前端`JSON.parse`直接报错，整个分析白跑 | 后端解析时做宽松处理（比如先用正则去掉可能的markdown代码块标记再解析），解析失败时返回友好的错误状态而不是裸异常 |
| 成本失控却没人发现 | prompt写错导致每次调用的输出token暴增（比如AI开始"复读"或者陷入啰嗦），一两周后账单异常才发现 | 加每日token消耗的简单统计，哪怕只是写进日志，定期手动看一眼，比完全没监控强 |

### 任务 1.1：接入 NewsAPI 替代 Playwright 爬虫

**目标**：解决 CoinDesk 爬虫超时、成功率低的问题。

**执行步骤**：

```
给 Trae 的指令：

1. 访问 https://newsapi.org 注册账号，获取免费 API Key
   （免费版限制：100次请求/天，每次最多返回100条，这点要写进代码注释提醒未来的自己）

2. 在项目根目录的 .env 文件中新增：
   NEWS_API_KEY=你的key

3. 修改 app/tasks.py 中的 crawl_crypto_news 函数：
   - 优先调用 NewsAPI 的 /v2/everything 端点
   - 查询参数：q=bitcoin OR cryptocurrency OR ethereum, language=en, sortBy=publishedAt
   - 获取结果后翻译为中文（复用项目里已有的通义千问调用逻辑做翻译，不要新引入翻译库）
   - 如果 NewsAPI 调用失败（403/429/超时），自动降级到原有 Playwright 爬虫逻辑
   - 降级逻辑要打日志，方便后续判断是不是配额用完了

4. 跑一次单元测试：手动调用 crawl_crypto_news，确认能拿到结构化新闻数据
```

**可能踩的坑 + 最优解**：

| 风险点 | 具体表现 | 解决方案 |
|---|---|---|
| 免费配额用尽 | 100次/天，如果设了定时任务+手动测试混着跑，很容易当天就超 | 代码里加一个简单的请求计数器（存 Redis，按天重置），超过 80 次自动停止调用 NewsAPI、直接走降级爬虫，避免 429 报错刷屏 |
| NewsAPI 免费版不让用于商业生产环境 | 条款里写的是"开发测试用"，如果以后要做成产品对外提供服务，这是违反 ToS 的 | 现在个人用没问题，但如果后续要给别人用/收费，必须提前换成付费版或者换 CryptoPanic 等允许商用的接口。**现在就把这行注释写进代码**，不要等违规了才想起来 |
| 翻译质量不稳定 | 复用通义千问做翻译，可能把币种名/专业术语翻译错（比如把"FOMO"翻译成奇怪的中文） | 翻译 prompt 里明确要求"专业术语、币种代码、人名保留英文原文，只翻译普通叙述部分"，并在 prompt 里给2-3个示例 |
| 降级逻辑没有真的被测试到 | 写了降级代码，但从来没真的触发过，等到 NewsAPI 真出问题时降级逻辑本身是错的 | 故意把 API Key 改成错的，跑一次，确认真的会触发 Playwright 降级且不报错崩溃 |

### 任务 1.2：配置定时任务 + 飞书 Webhook 通知

**目标**：每天 8:00 / 20:00 自动跑 BTC+ETH 分析，极端情绪自动推送飞书。

**执行步骤**：

```
给 Trae 的指令：

1. 修改 app/celery_app.py，增加 Celery Beat 配置：
   - 任务名：daily_sentiment_analysis
   - 执行时间：每天 08:00 和 20:00（注意服务器时区，确认是不是UTC，
     如果服务器是UTC时区，国内8点要写UTC 0点，20点要写UTC 12点）
   - 任务内容：依次跑 BTC 和 ETH 的情绪分析

2. 在飞书群里添加"自定义机器人"，获取 Webhook URL
   （飞书群设置 → 群机器人 → 添加机器人 → 自定义机器人）

3. 将 Webhook URL 存入 .env：
   FEISHU_WEBHOOK_URL=你的webhook地址

4. 新建 app/notifier.py：
   - 封装 send_feishu_alert(coin, sentiment_score, summary) 函数
   - 当 sentiment_score < 0.35 或 > 0.75 时触发推送
   - 消息内容用飞书的"消息卡片"格式（不要发纯文本，太丑），
     包含：币种、情绪分数、一句话摘要、当前价格、F&G指数

5. 在定时任务执行完分析后，调用 notifier 判断是否需要推送
```

**可能踩的坑 + 最优解**：

| 风险点 | 具体表现 | 解决方案 |
|---|---|---|
| 时区搞错，任务在凌晨3点跑 | Docker 容器默认时区通常是 UTC，"每天8点"如果直接写本地时间会全部错位 | 在 celery_app.py 里明确设置 `timezone = 'Asia/Shanghai'`，Celery Beat 支持指定时区，不要用UTC手动换算，容易算错且未来改时间麻烦 |
| 飞书机器人被风控/频率限制 | 飞书自定义机器人有频率限制（通常5次/分钟），如果代码有bug导致重复触发会被临时封 | 加一个"同一币种同一信号15分钟内不重复推送"的简单去重逻辑（用Redis存最后推送时间） |
| Celery Beat 和 Worker 没有同时启动 | 配置了 Beat 任务但忘记在 docker-compose.yml 里加 beat 服务，定时任务永远不会触发 | 检查 docker-compose.yml，确认有单独的 `celery beat` 服务在跑，不是只跑了 `celery worker`。这是最容易漏的一步 |
| 推送消息里的数字看不出趋势 | 只发"BTC情绪分数0.28"，用户不知道这意味着什么 | 消息卡片里加一句AI生成的人话总结，比如"市场出现极度恐慌情绪，可能存在超卖机会，但需结合更多信号判断" |
| 任务执行失败没人知道 | 定时任务本身挂了（比如API超时），但因为是后台静默执行，没人发现 | 给 Celery 任务加 try/except，失败时也发一条飞书消息："今日8点定时分析失败，原因：xxx"，失败也要让你看到 |

## 阶段二：P1 任务（验证阶段一稳定后再做）

### 任务 2.1：OKX 合约 API 替代 Binance

```
给 Trae 的指令：

1. 注册 OKX，申请 API Key（只需要只读权限，不要给交易权限）
2. 找到原代码中调用 Binance 合约接口（返回451）的位置
3. 替换为 OKX 的 /api/v5/public/open-interest 和资金费率接口
4. 注意 OKX 的币种代码格式和 Binance 不同（比如 BTC-USDT-SWAP vs BTCUSDT），
   做好映射转换
5. 旧的 Binance 调用代码不要删，注释掉或者做成可切换的数据源配置，
   万一以后能解锁还能用
```

**坑点**：OKX 接口字段命名和 Binance 不一样，**一定要先用 Postman 或 curl 单独测试一次接口返回的真实数据结构**，别直接照着Binance的字段名硬套，会取错字段。

### 任务 2.2：多币种概览仪表盘

```
给 Trae 的指令：

1. 后端新增 /api/overview 端点，并行（用 asyncio.gather 或 Celery group）
   获取 BTC/ETH/SOL 的情绪分数+24H涨跌幅+F&G指数
2. 前端新增 OverviewDashboard.vue 组件，卡片式横向排列三个币种
```

## 阶段三：P2 任务（功能完善，时间充裕再做）

### 任务 3.1：历史预测追踪与胜率回测

```
给 Trae 的指令：

1. 新建数据表 prediction_history：
   字段包括 coin, bias(多/空), confidence, predicted_at, 
   price_at_prediction, price_after_Nh（定时任务后续回填）, is_correct
2. 每次AI分析产出bias结论时，自动写入这张表
3. 新增定时任务，N小时后（比如24小时）回填实际价格，
   对比判断当时的预测是否正确
4. 新增 /api/track-record 端点，返回历史胜率统计
```

**坑点（这个任务的坑比想象中多，要重点提醒）**：

- **"正确"的定义要提前想清楚**：AI说"看多"，如果价格涨了0.1%算对吗？建议设一个最小阈值（比如涨跌超过1%才算validated，否则标记为"中性/不计入胜率"），否则胜率数字会失真
- **不要让这个功能反而误导你**：如果胜率长期低于50%，说明AI的判断不比随机猜硬币强，这时候不要急着调prompt"让胜率好看"，而是要诚实面对——可能这个分析维度本身就不够，需要加更多数据源（这也是为什么文档里链上数据被列为重要优化方向）

### 任务 3.2：用户系统（可跳过）

文档里已经说了"仅个人使用可跳过"，**我的建议是真的跳过**，除非你明确要把这个做成对外产品。个人项目加用户系统纯粹是增加维护成本，没有实际收益。

## 技术债务处理建议（不算紧急，但建议跟着P1一起做）

| 债务 | 建议处理时机 | 原因 |
|---|---|---|
| Playwright 启动慢(3-5s) | 任务1.1做完后这个问题已经大幅缓解（NewsAPI优先），可以先不处理 | 优先级低，等到Playwright完全只是降级方案时再考虑要不要换httpx+BS4 |
| Celery 单 Worker | 完成阶段一之后立刻处理 | 一旦上了定时任务+多币种并行请求，单Worker会成为瓶颈，建议至少配2-3个worker |
| 数据库在 Docker Volume 里 | 在做历史预测追踪（任务3.1）之前必须处理 | 一旦开始存历史数据，数据持久性变得关键，Docker Volume 有被误删的风险，迁移成本会越往后越高 |

## 每个任务的明确验收标准（不要只靠"感觉跑通了"）

"本地测试通过"这句话太模糊，Trae 可能理解成"代码跑起来没报错"就算过，但实际输出可能是错的。每个任务建议按下面的清单逐项打勾，而不是粗略过一遍：

**任务1.0（AI引擎安全带）验收**：
- [ ] 故意断网或改错API地址，确认会重试2次后才报失败，不是立刻崩
- [ ] 手动构造一个格式错误的假返回（比如带markdown代码块的JSON），确认解析逻辑能处理，不会让整个请求500
- [ ] 检查日志/Redis里能看到token消耗记录

**任务1.1（NewsAPI）验收**：
- [ ] 正常情况下能拿到真实新闻数据，且翻译后中文术语没有明显错误
- [ ] 故意填错API Key，确认自动降级到Playwright且全程无未捕获异常
- [ ] 查看日志，确认请求计数器在工作（连续调用多次后能看到计数变化）

**任务1.2（定时+飞书）验收**：
- [ ] 手动把定时任务时间改成"1分钟后"触发一次，确认真的会跑
- [ ] 手动伪造一个情绪分数0.2的测试数据，确认飞书真的收到推送且卡片格式正常显示（不是纯文字刷屏）
- [ ] 连续触发3次"极端值"，确认15分钟内的去重逻辑生效，没有刷屏
- [ ] 故意让分析任务内部抛异常，确认飞书会收到"任务失败"的提示，而不是静默无消息

## 出问题时的回滚预案

| 场景 | 怎么快速止血 |
|---|---|
| 飞书半夜疯狂推送 | 改 `.env` 里 `ENABLE_AUTO_NOTIFY=false`，重启服务（不需要改代码、不需要等commit） |
| 定时任务跑出来的分析结果持续异常 | 先去 Celery Beat 配置里临时注释掉这个任务，而不是去改AI prompt"救场"——先止血，再排查 |
| NewsAPI 配额被意外用尽导致全部降级到慢速Playwright | 检查 Redis 里的计数器有没有重置逻辑出bug，临时手动清空计数器key作为应急 |
| 怀疑某次commit引入了问题但不确定是哪个 | 这就是为什么阶段零要记录"基准状态"和铁律2要求每个任务单独开分支——用 `git bisect` 或者直接挨个分支测试，能比"盯着代码看"快很多 |

## 给 Trae 的完整执行顺序总结

```
Day 0: 阶段零（环境验证，15分钟）→ 记录基准状态
Day 1: 任务1.0（AI引擎安全带）→ 验收清单逐项打勾 → commit
       任务1.1（NewsAPI）→ 验收清单逐项打勾 → 跑1天观察日志 → commit
Day 2: 任务1.2（定时+飞书）→ 验收清单逐项打勾（重点测试回滚开关）→ commit
Day 3-4: 观察阶段一稳定运行后，做任务2.1 + 2.2
之后: 看时间和兴趣，做阶段三

每个任务做完后必须做的事：
1. 更新 README.md 里对应功能的说明
2. 检查 .env.example 文件是否同步加上了新的环境变量占位（不要带真实值）
3. 单独commit，commit message 写清楚改了什么、为什么改
```
